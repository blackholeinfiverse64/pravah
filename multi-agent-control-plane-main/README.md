# Pravah: Multi-Agent Control Plane (Converged Edition)

Pravah is a unified autonomous control plane that integrates real-time system monitoring, deterministic metric-based decisions, strict policy governance, and authenticated action execution.

---

## 1. System Topology & Flow

The system operates in a unidirectional feedback loop across four decoupled components:

```text
+-------------------+      Telemetry      +----------------------+
|  Monitor Service  | ------------------> | FastAPI Control Plane|
|   (Port 5004)     |   (Runtime-Ingest)  |     (Port 8000)      |
+-------------------+                     +----------------------+
         ^                                           |
         | Polls Health                              | Authenticated Request
         |                                           v
+-------------------+                     +----------------------+
|  Target Service   | <------------------ |   Rayyan Executor    |
|   (Port 5001)     |    Executes Action  |     (Port 5003)      |
+-------------------+                     +----------------------+
```

### Flow Breakdown (1-Page Summary)
1. **Telemetry Ingest (`/control-plane/runtime-ingest`):** Telemetry is POSTed to the Control Plane backend. The stage is logged as `detection`.
2. **Decision Engine:** Telemetry metrics (floats scaled up by `100`) are mapped to a `DecisionRequest` and evaluated against threshold policies (e.g. CPU >= 90). The stage is logged as `payload_emitted`.
3. **Governance Check:** Eligibility rules (e.g. action constraints in `DEV`, `STAGE`, `PROD`) are checked. Active cooldowns (120s for scale actions) and repetition checks are thread-safely verified using a local persistent state file (`logs/control_plane/governance_state.json`).
4. **Execution Handoff:** If governance admits the action, the Control Plane signs the request with HMAC SHA256 and dispatches it to the Rayyan Executor. The stage is logged as `action_received`.
5. **Real-world Execution:** The Executor validates the request signature, executes the OS level docker/k8s action (e.g., container restart), verifies container status, and returns the verified execution result. The stages are logged as `execution_result` and `verification`.

## 2. Integration Endpoints

### Ingestion API (Control Plane — Port 8000)
* **Endpoint:** `POST /control-plane/runtime-ingest`
* **Telemetry Payload:**
  ```json
  {
    "service_id": "web1",
    "timestamp": "2026-06-03T20:50:55Z",
    "status": "degraded",
    "metrics": {
      "cpu": 0.95,
      "memory": 0.83,
      "error_rate": 0.0,
      "uptime": 12345
    },
    "issue_detected": true,
    "issue_type": "cpu_spike",
    "recommended_action": "scale_up"
  }
  ```

### Execution API (Rayyan Executor — Port 5003)
* **Endpoint:** `POST /execute-action`
* **Authentication:** Signed headers generated by `build_signed_headers` (`X-Service-Signature`, `X-Service-Timestamp`, `X-Service-Nonce`, `X-Service-Id`).

## 3. How to Run the Full Loop (Quick Start)

### Step 1: Initialize Python Environment
From the repository root directory:
```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

### Step 2: Spin Up Services (Separate Terminals)

#### Terminal 1: FastAPI Control Plane Backend
```powershell
.\.venv\Scripts\python.exe -m uvicorn control_plane.backend.app.main:app --port 8000
```

#### Terminal 2: Rayyan Executor Service
```powershell
$env:PYTHONPATH="."
.\.venv\Scripts\python.exe reliability-controller2-main/executer/app.py
```

#### Terminal 3: Target Service (`web1`)
```powershell
.\.venv\Scripts\python.exe reliability-controller2-main/web1/app.py
```

#### Terminal 4: Monitor Service
```powershell
.\.venv\Scripts\python.exe reliability-controller2-main/monitor/app.py
```

## 4. Run the E2E Loop Test
To run a full loop test automatically (simulating degradation, monitor detection, Control Plane ingestion, decisioning, governance, signed execution, and trace log verification):
```powershell
.\.venv\Scripts\python.exe e2e_integration_test.py
```
This script triggers a failure on `web1`, waits for the monitor to capture the CPU spike, forwards it to the Control Plane ingest, and outputs the resulting sequence validation trace in `trace_log.jsonl`.
